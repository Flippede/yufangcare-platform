# 御方通和自动佣金与门店结算体系 V1

## 1. 最终业务口径

本阶段将套餐 15%/25%/60% 奖励和普通商城佣金统一接入受限佣金账本，但不建设 B1 提现。B1 是订单佣金的结算接收方：佣金先进入未结算金额，再由总部按结算周期生成批次，未来通过微信分账完成后转为已结算金额。

C1 与 B1 之间仍是线下结算。C1 向永久归属 B1 提交结算申请，由该门店的店长或店员查看并确认线下完成，总部不参与该正常流程。

佣金不写入 CRMEB `now_money`、积分、旧分销佣金、钱包或提现系统，不能用于商城、套餐或采购支付。本阶段不改写 CRMEB 商品、订单、支付和退款状态机。

## 2. 佣金产生

### 2.1 普通商城

- 仅接收真实已支付、未删除、未取消的普通商品主订单。
- 规则按“指定商品 > 指定分类 > 全部商品”选择，冻结 C1/B1 比例、规则版本和观察天数。
- 观察期为 0 天时在订单完成事件中立即入账；非 0 天时形成 `observing` 应计，由 `yfth:commission-settle` 到期处理。
- C1 与 C2 必须具有有效一级推荐关系且共同归属同一个 B1；没有有效 C1 时只计算符合规则的 B1 佣金。
- 同一订单、应计和流水均有唯一约束，重复事件不会重复入账。

### 2.2 套餐 15%/25%/60%

- 复用版本化套餐奖励快照和第 1/2/3 个有效会员序号。
- 奖励只进入 C1 佣金账户，同时计入责任 B1 的未结算责任金额。
- C2 成为永久会员后关闭 active 推荐关系；历史快照和已发生流水保持不变。

## 3. B1 未结算与已结算

`yfth_store_commission_account` 以 `store_id` 为主体，仅形成以下业务投影：

- `unsettled_cent`：订单佣金已经产生，但尚未通过总部结算周期完成结算。
- `settled_cent`：已经进入总部结算批次并收到成功分账/结算回调的金额。
- `c1_pending_cent`：本店 C1 尚未完成线下结算的责任金额。
- `c1_paid_cent`：本店已确认线下完成的 C1 结算金额。
- `reversed_cent`：退款等业务导致的累计冲正金额。

B1 页面只展示未结算、已结算和结算明细。不存在 B1 自有余额、C1 代付余额、可提现金额、提现申请、金额冻结、提现成功或 FIFO 提现分配。

## 4. 总部结算批次

结算链路固定为：

`订单佣金 -> 结算周期 -> B1 结算批次 -> 微信分账适配器 -> 回调记录 -> 已结算`

批次状态：

- `pending`：待结算；
- `processing`：结算中；
- `settled`：已结算；
- `exception`：异常，可在原因处理后重新进入结算中。

每条门店佣金流水只能属于一个结算批次。批次金额由周期内未分配的不可变流水净额计算，B1 不能输入或申请金额。成功回调才将门店投影从未结算转为已结算；失败回调不移动余额。

## 5. 微信分账预留

本阶段不调用真实微信分账，但数据模型已经冻结以下边界：

- B1 分账接收方类型、加密账号、脱敏账号和加密名称；
- 微信商户主体加密快照；
- 结算金额、周期和内部批次号；
- 微信批次号、明细号、处理状态和异常原因；
- 幂等回调事件、脱敏回调内容和回调时间；
- 独立分账回退编号、原批次、回退金额、微信回退单号、状态、原因和回调事实；
- 批次与原始佣金流水的一对多不可变分配关系。

接收方敏感字段由环境密钥加密。B1 端不维护接收方，总部在既有财务/结算页面配置。未来接入真实适配器时只替换“开始结算”的外部调用，不重构账户、批次或回调表。

## 6. C1 线下结算

C1 页面展示应结算佣金、已申请结算、已完成结算及明细。提交时服务端从权威永久归属解析责任 B1，客户端不能选择 `store_id`。

申请后金额从 C1 可用转为冻结，但本店 C1 待处理总额不重复增加。店长或店员完成本店线下付款记录后，冻结金额转为累计完成并同步更新门店 C1 待处理/已完成投影。重复申请和重复完成均幂等，跨店处理 fail closed。

## 7. 退款冲正

- 观察期内全额退款取消应计，不入账。
- 已入账退款追加关联原流水的负数流水，不覆盖原事实。
- C1 佣金和 B1 未结算金额按原快照冲正，重复退款通知不重复冲正。
- 已完成结算批次不伪造回滚；后续负数流水进入下一结算周期净额。
- 未来真实微信分账回退使用独立回退事实表，不覆盖原结算批次或原始佣金流水。
- 余额可因冲正形成负数，未来收入先抵扣；负数 C1 不能发起结算。

## 8. 权限与页面

- C1：只读本人账户、流水和结算记录，可以向责任 B1 提交结算申请。
- B1：在工作台“结算”查看本店未结算、已结算、批次明细和 C1 申请；店长/店员按现有门店权限确认 C1 线下完成。
- 总部：在既有财务/结算页面维护接收方、生成结算批次、启动预留分账流程、查看回调和异常。
- 前端身份和门店参数不是授权依据；服务端继续校验用户 Token、角色和门店范围。

不新增顶级菜单，不提供任何 B1 提现 API 或页面。
