# YFTH Release Scope - Headquarters Mall, Store Attribution, Membership And Referral Rewards

> 文档状态：`Headquarters Mall Release Scope - Frozen Draft`
>
> 最新上线范围口径已冻结，旧的 B 商家独立商城方案已被取代；尚未允许直接开始业务代码开发。收益比例、退款边界和部分状态门禁仍需开发前确认，后续开发必须以本文件为最新产品依据。

## 1. 当前上线总原则

- 小程序 C 端始终只显示总部统一商城。
- 现阶段不建设 B 商家独立商城、门店独立首页、独立商品、独立装修或独立线上价格体系。
- 保留 CRMEB 原有总部商城、商品、SKU、购物车、订单、支付、退款、物流、售后和页面装修能力。
- 所有 C 端线上普通商品订单均由总部商城产生，货款进入总部公户对应的总部支付主体。
- 总部商城订单继续使用 CRMEB 原订单、支付、退款、物流和售后主链路。
- B 商家不负责总部商城消费者订单的线上收款，也不拥有自己的线上商城页面。

## 2. C 端首次注册和未绑定状态

- C 用户首次注册时默认不绑定任何 B 商家。
- 未绑定用户可以正常浏览和购买总部商城商品，其订单均属于总部自然订单。
- 未绑定用户的历史订单不产生 B 商家收益，也不产生 C 端直推消费分成。
- 用户以后绑定 B 商家时，不追溯绑定前的历史订单和收益。
- 仅浏览总部商城、品牌内容或门店资料不能自动绑定 B 商家。

## 3. B 商家线上功能范围

现阶段 B 商家仅拥有：

- 总部商城首页入口和 B 端工作台。
- 总部采购台，以及复用 YFTH 供应链与门店库存的采购能力。
- 客户归属和层级数据。
- 会员码管理和会员激活记录。
- 线下套餐成交确认。
- 总部商城绑定客户消费数据。
- B 商家收益台账和 C 端直推子分配台账。
- 结算申请、凭证和状态。

现阶段 B 商家不拥有：

- 独立商城首页、独立商城装修或独立线上商品页面。
- 独立消费者线上收款。
- 在线展示或销售 `9800` 会员。
- 在线展示或销售 `9800` 业务套餐。
- 总部商城消费者订单的发货职责或独立售后职责。

B 商家向总部采购继续复用 YFTH 供应链与门店库存能力，但不得与总部消费者商城订单混用订单、库存或结算链路。

## 4. 永久 B 商家归属

C 用户只能通过下列正式业务事件形成 B 商家永久归属：

1. 有效 C 端一级直推绑定。
2. 扫描 B 商家一次性动态会员码并成功激活会员。
3. 未绑定用户线下购买业务套餐后，由 B 商家完成真实线下套餐成交确认。

归属规则：

- 一旦形成 active B 商家归属，B 商家正常经营期间永久有效。
- 用户不能主动解绑、换店或切换到其他 B 商家线上归属。
- 已绑定 B1 的用户扫描 B2 会员码必须拒绝，B2 不得覆盖 B1 归属。
- 用户在其他 B 商家发生线下消费，不改变线上系统归属。
- B 商家不得私自转移客户，非授权总部人员不得修改客户归属。
- 同一 C 用户同一时刻最多存在一个 active B 商家归属。
- 历史归属必须保留，不得物理删除。
- 前端始终显示总部商城，不因 B 商家归属切换商城页面。

B 商家归属必须与以下对象分离：

- 会员资格。
- 业务套餐记录。
- C 端推荐层级。
- B 商家经营角色。
- CRMEB 用户等级和 CRMEB 付费会员。
- CRMEB 分销关系。
- CRMEB 订单备注。

## 5. C 端一级直推关系

- 只有正式会员拥有一级直推资格，普通用户没有直推资格。
- C1 是 B1 正式会员时可以推荐 C2；C2 通过 C1 的有效推荐关系进入系统后，永久归属 B1。
- C2 在未成为会员前处于 C1 的 active 一级直推层级下。
- 只做一级直推，不做多级分销、团队收益、跨代收益或间接奖励。
- C1 不获得 C2 推荐客户的奖励，只获得 C2 本人在 active 层级期间产生的符合规则的奖励。

active 一级推荐关系有效期冻结为：

- 不采用现有兼容推荐候选的 90 天有效期，不设置固定天数自动到期，也不因长期未消费自动关闭。
- C2 尚未成为会员，且关系未因风控、无效绑定或总部受控纠错关闭时，持续处于 C1 active 一级层级。
- C2 重复购买业务套餐期间关系继续有效，每次有效套餐成交均可为 C1 生成奖励候选。
- C2 购买会员时，该次会员成交仍属于 C1 的有效推荐事件；会员激活成功后，C1 -> C2 active 关系立即关闭且不自动恢复。
- 历史推荐关系、成交、奖励、序号和结算记录持续保留。

C2 成为会员时：

- C2 的会员购买可为 C1 生成一笔循环三三制奖励候选事件。
- 会员激活成功后，C2 立即脱离 C1 的 active 推荐层级，成为 B1 商家的直属会员，并取得自己的一级直推资格。
- C1 与 C2 不再存在持续上下级关系，只保留历史推荐、成交、奖励、冲正和结算记录。
- C1 不再获得 C2 后续总部商城消费分成、业务套餐购买奖励、会员或其他业务成交奖励。

## 6. 会员产品

会员和业务套餐是两个独立产品，不得共用一个业务对象或状态机。

会员当前规则：

- 当前价格为 `9800` 元，由 B 商家线下收款；资金不进入总部商城订单或总部公户。
- 当前默认永久有效，但数据模型和接口必须预留一个月、一年、自定义有效期和永久有效。
- 会员福利接口须预留总部商城会员价、折扣、优惠券、赠品、积分倍率、身份福利和其他配置化权益；具体福利后续配置，不在本文写死。
- 会员身份赋予一级直推资格。
- 会员不等于业务套餐，购买会员不会自动产生业务套餐权益。
- 已经是 active 永久会员的用户不得再次购买或激活会员；永久会员第一版不支持续费。月度、年度和自定义期限仅保留未来扩展接口，不在第一版开放。

会员激活流程：

1. B 商家线下确认收到会员款，由 `franchisee` 或 `store_manager` 创建 pending 会员成交并登记付款方式、收款时间及凭证。
2. 系统生成与该 pending 成交绑定的一次性动态会员码。
3. C 用户扫码页面明确展示 B 商家、会员价格 `9800`、永久会员、会员福利和直推资格，并由本人确认已线下付款。
4. C 用户确认即构成 C 端本人确认，不再增加扫码后的第二次人工确认步骤。
5. 服务端校验成交、码、门店归属、当前用户和幂等状态；校验通过后会员立即生效。
6. 会员码短时有效、单次使用、门店隔离、服务端存哈希、可审计且幂等；重复请求返回原结果。
7. 已绑定其他 B 商家或已是 active 永久会员的用户必须拒绝。

会员激活必须在一个事务内按以下原子顺序完成：

1. 锁定 pending 会员成交。
2. 锁定 C2 当前 active 推荐关系。
3. 固化会员成交快照。
4. 若存在 C1，先创建并唯一化本次会员成交奖励候选。
5. 关闭 C1 -> C2 active 推荐关系。
6. 创建并启用 C2 会员实例。
7. 赋予 C2 一级直推资格。
8. 提交事务。

任一步失败必须整体回滚，不得出现会员已生效但关系未关闭、关系已关闭但奖励候选丢失、重复会员实例或重复奖励候选。

## 7. 9800 业务套餐

- 业务套餐当前价格为 `9800` 元，是正常线下业务套餐，不是会员。
- 业务套餐由 B 商家线下收款，资金不进入总部商城订单或总部公户。
- 套餐不会自动赋予会员身份或直推资格。
- 同一非会员用户可以重复购买套餐。
- 同一 C2 仍未成为会员且仍处于 C1 active 直推层级时，每次有效套餐成交都可以为 C1 生成一笔三三制奖励候选事件。

线下套餐成交确认流程（历史“套餐核销”术语仅指成交确认，不代表服务权益消费）：

1. C 用户向 B 商家线下购买 `9800` 业务套餐。
2. C 用户在小程序确认“已线下购买 9800 套餐并生成成交码”，页面展示 B 商家、套餐名称、价格和确认声明。
3. C 用户确认后生成短时、单次、哈希存储的动态成交确认码；该码只确认本次线下成交，不代表服务权益已消费。
4. `franchisee` 或 `store_manager` 使用 B 端工作台扫码，填写付款方式、收款时间、收据编号或上传支付凭证。
5. 服务端校验 C 用户、B 商家归属、active 推荐关系、操作权限和幂等键，成功后创建独立线下套餐成交记录。
6. 每次重复购买必须创建新的成交编号和成交快照；同一动态成交码只能成功创建一笔成交。
7. 未绑定用户在成交确认后永久绑定该 B 商家；无 active 直推人时不产生三三制奖励，有 active 直推人时生成奖励候选。
8. 不创建 CRMEB `store_order`，不复用预约或服务权益核销表，不自动赋予会员或直推资格；套餐服务履约和权益消费属于后续独立状态机。

套餐成交确认和会员码必须使用不同业务场景、不同权限和不同操作接口，不得复用同一业务码。

## 8. 线下可信成交与角色权限

### 8.1 允许角色与能力

只有 `franchisee` 和 `store_manager` 且具备服务端能力 `offline_membership_sale` 或 `offline_package_sale` 时，才可登记相应线下成交。

默认禁止 `store_staff`、普通 C 用户、`service_mentor`、城市合伙人、其他门店角色，以及缺少对应能力的加盟商或店长。`store_staff` 第一版只可查看必要状态，不得生成会员码、登记线下收款、确认套餐成交、提交退款、登记奖励结算或修改会员/推荐关系。

总部管理员使用 admin-token 和独立后台 API 权限处理审核、异常、退款和冲正，不得复用门店 user-token 接口。

### 8.2 通用线下成交记录

每笔会员或套餐成交必须先创建独立且不可变的线下成交记录，至少保存：

- 服务端生成的成交编号和 `business_type`（`membership` / `package`）。
- `store_id`、C 用户 UID、当前 active 推荐人 UID（可空）和当前 B 商家归属快照。
- 成交金额快照（当前固定 9800）、付款方式、B 商家确认收款时间。
- 线下收据编号或私有支付凭证。
- 操作人 UID、操作角色、幂等键、创建/确认时间、状态、规则版本和审计信息。

凭证和信任门禁：

- 至少提供一个可追踪凭证：线下收据编号或私有支付凭证上传；不得完全无凭证创建高额奖励型成交。
- 凭证仅允许私有访问，跨店账号不得读取。
- 前端传入的 `store_id`、`referrer_uid`、`amount`、`status` 和 `rule_version` 均不是可信事实，必须由服务端重新解析和校验。

### 8.3 总部抽查和异常控制

- 总部可按门店、操作人、时间、金额和异常频率抽查成交，冻结异常成交和奖励候选，并要求补充凭证。
- 总部不得物理删除成交；异常关闭须保留原因、操作人和审计。
- B 商家不得通过反复创建、删除或重建成交改变循环三三制序号。

## 9. 会员成交与套餐成交共用循环三三制

会员成交和业务套餐成交进入直推人的同一条循环奖励序列。例如：

| 有效成交序号 | 成交示例 | 比例 | 当前 9800 基数金额 |
| ---: | --- | ---: | ---: |
| 1 | 套餐 | 15% | 1470 |
| 2 | 会员 | 25% | 2450 |
| 3 | 套餐 | 60% | 5880 |
| 4 | 会员 | 15% | 1470 |
| 5 | 套餐 | 25% | 2450 |
| 6 | 套餐 | 60% | 5880 |

循环规则：

- 第 1、4、7、10……笔为 15%。
- 第 2、5、8、11……笔为 25%。
- 第 3、6、9、12……笔为 60%。
- 每三笔形成一个循环，循环可以持续进行，第 4 笔重新从 15% 开始。
- 仍然只计算一级直属关系；循环不代表多级分销。

计算基数和历史规则：

- 当前会员成交和业务套餐成交基数均为 `9800`。
- 当前 15% / 25% / 60% 档分别为 1470 / 2450 / 5880。
- 后续价格变化时必须使用成交快照和规则版本，不得按当前价格回算历史。

付款责任和单笔账务口径：

- 会员和业务套餐均由发生该笔线下成交的 B 商家销售并收款，该 B 商家是循环三三制奖励的应付主体。
- B 商家线下向 C 端直推人付款；总部不承担、补贴或直接支付该奖励。
- 系统只记录成交、奖励候选、序号、应付金额、观察期、冲正、结算凭证和状态。
- 每笔成交使用自身金额快照计算。当前 9800 基数下：15% 奖励 1470，B 留存 8330；25% 奖励 2450，B 留存 7350；60% 奖励 5880，B 留存 3920。
- 三笔成交总收入为 29400，奖励合计 9800，B 商家合计留存 19600。15% + 25% + 60% 等于一个 9800 基数，不是把三笔 29400 全部返出。

```text
B 商家该笔成交收入 = B 商家该笔留存金额 + C 直推人三三制奖励
```

不得把三三制记为总部应付，不得与总部商城普通商品的 B 收益/C 子分配混为一套台账，不得从总部商城订单资金重复支付，也不得写入 CRMEB 余额、佣金、分销、积分或提现字段。

B 商家支付奖励后必须登记或上传 B -> C 线下结算凭证。C 端可查看待观察、待结算、已结算和已冲正状态；总部可抽查、审核异常和执行冲正，但不是默认付款人。

奖励序列归属于直推人 C1：

- 会员和套餐事件进入 C1 的同一个序列，按有效成交事件计数，不按唯一客户人数计数。
- 同一 C2 在未成为会员前重复购买套餐，每次有效成交都可以推进 C1 的序号。
- C2 购买会员时也推进 C1 序号。
- C2 会员生效并脱离 C1 后，C2 后续成交不再推进 C1 序号。

## 10. 三三制观察期、序号和冲正

- 会员激活或线下套餐成交确认后先生成待观察奖励候选。
- 观察期配置范围为 7–15 天，默认具体天数待确认。
- 观察期内退款、撤销、异常或被判定无效时，不占最终序号。
- 观察期通过后，在事务内分配正式序号。
- 多笔事件同时通过时必须使用确定性排序和数据库锁，不能依赖请求先后。
- 正式序号分配后发生退款或撤销时生成冲正记录。
- 已冲正序号不释放、不补位、不重排，后续新事件继续使用下一个序号。
- 奖励历史、规则版本、金额快照、事件来源和冲正记录不得删除。
- 观察期继续配置为 7–15 天；默认天数待项目主控确认。该参数待定不阻塞需求文档合并，但阻塞真实奖励解冻和上线，开发人员不得自行选择默认值。

## 11. 线下退款和撤销

会员费和套餐费由 B 商家线下收取，总部不能依赖 CRMEB 支付回调自动判断退款。

- B 商家提交会员或套餐退款/撤销申请，登记凭证、金额、原因和时间。
- B 商家不能直接删除成交或奖励。
- 总部后台审核退款/撤销申请；拒绝时记录原因。
- 审核通过后，观察期内候选失效且不占序号；已分配奖励生成冲正，原序号不释放、不补位、不重排；原成交和奖励记录保留；新增不可变退款/冲正记录。
- 全程记录操作人、门店、用户、金额、前后状态、时间、IP、凭证和审计。
- 会员全额退款生效后，会员身份失效且 C2 失去一级直推资格；C2 已建立的下级关系处置留待未来专项。
- 会员退款不恢复原 C1 -> C2 active 推荐关系，该关系保持历史关闭；C2 仍保留原 B 商家普通客户归属，除非 B 商家终止。
- 套餐全额退款不改变会员状态、永久 B 商家归属或 active 推荐关系。
- 第一版不开放会员或套餐自动部分退款状态机。线下部分退款由 B 商家提交总部异常审核，不自动撤销会员、改变 active 关系或回收/重排序号；是否按比例冲正须由总部异常调整明确执行。正式开放部分退款前必须新增规则版本并重新架构审核。

## 12. 总部商城订单的 B 商家收益和 C 端消费分成

未绑定用户：

- 总部商城订单不产生 B 商家收益或 C 端消费分成。
- 用户以后绑定商家时不追溯历史订单。

已绑定 B1 的用户：

- 总部商城订单由总部收款，并按配置规则计算 B1 的商家总收益。
- 若购买用户仍处于 C1 的 active 直推层级，从 B1 商家总收益中拆分 C1 应得消费分成。
- 总部只向 B1 结算一笔完整的 B 商家总收益；B1 再线下向 C1 结算。
- 总部不得再额外向 C1 支付第二笔钱。

```text
B 商家总收益 = B 商家实际留存收益 + C 端直推人应得分成
```

系统必须分别记录：

- B 商家总收益台账。
- C 端直推子分配台账。
- B 商家实际留存金额。
- 总部向 B 商家结算状态和凭证。
- B 商家向 C 端直推人结算状态和凭证。
- 退款和售后冲正。

必须防止同一订单由总部重复向 B、C 分别支出，防止把 C 端分成记为总部直接应付，防止混淆 B 商家总收益与净收益，也不得写入 CRMEB 余额、佣金或分销字段。

B 商家总收益比例和 C 端消费分成比例均待确认，必须配置化并保留规则版本。订单成交时保存规则和金额快照，不信任前端传入比例。

比例待定不阻塞需求文档合并，但阻塞总部商城收益阶段的真实金额计算、结算功能和上线；开发人员不得自行设置默认业务比例。

C2 成为会员并脱离 C1 后，B1 仍可获得 C2 后续总部商城订单对应的 B 商家收益，但 C1 不再获得 C2 的消费分成。

## 13. B 商家收益结算

本章的总部商城 B 商家收益结算，与线下会员/套餐三三制奖励是两套独立台账和责任链路：

- 线下会员/套餐三三制奖励由成交 B 商家以该笔线下成交收入承担，并由 B 商家直接线下结算给 C 端直推人。
- 总部不是三三制奖励付款人，不从总部商城订单资金支付或补贴该奖励。
- B 商家完成 B -> C 支付后登记私有凭证；总部仅执行抽查、异常审核和冲正治理。

- 总部商城订单支付成功后生成 B 商家待观察收益。
- 订单完成并经过退款/售后观察期后转为可结算。
- B 商家发起结算申请，总部审核后线下转账，登记凭证并标记完成。
- B 商家前端显示结算状态。
- 系统不自动打款，不写 CRMEB 余额、佣金、分销、`now_money`、`user_brokerage` 或 `user_bill`。

若 B 商家总收益中包含 C 端直推子分配：

- B 商家收到总部完整收益后，再线下向 C 端结算。
- B 商家登记 C 端结算凭证。
- C 端可查看待结算、已结算和冲正状态。

## 14. B 商家终止与就近接管

B 商家正式终止后，按以下优先级寻找可接管 B 商家：

候选范围以终止 B 商家门店自身的规范化行政区编码为基准，不按每个 C 用户收货地址分别计算。后续数据模型必须保存省、市、区县标准行政区 code；`system_store.address` 文本不能单独作为自动匹配依据。

1. 同区县其他可接管 B 商家。
2. 同市其他可接管 B 商家。
3. 同省其他可接管 B 商家。
4. 无可接管商家时进入无店铺绑定状态。

同一层级有多家候选时，系统提供候选列表，由总部人工选择，不随机自动分配，也不允许前端自行选择；必须二次确认、填写原因并审计。

接管规则：

- 原 C 端 active 直推层级保持不变，只迁移 B 商家归属。
- 迁移前历史订单和收益仍归原 B 商家历史台账；迁移生效后的新订单和收益归新 B 商家。
- 历史订单不追溯、历史收益不重算，历史会员、套餐、推荐、核销和审计记录全部保留。

无可接管商家时：

- 普通用户变为未绑定用户。
- 会员继续保留会员身份和总部提供的会员福利，但暂时没有 B 商家归属。
- 原历史关系继续保留。
- 现有 active C 端推荐关系可保留但进入暂停状态；暂停期间不产生总部商城 C 端消费分成，不允许通过该关系新增永久商家归属，也不允许生成新的线下会员或套餐成交。
- 总部商城新订单仅作为总部自然订单，不生成 B 商家收益或 C 端消费子分配。
- 后续接管成功后，仅对接管生效后的新订单计算新 B 商家收益；历史订单不追溯、不重算。

## 15. CRMEB 边界

明确禁止：

- 重写 CRMEB 商品、SKU、购物车、订单创建、支付回调、退款、物流和售后主流程。
- 删除 CRMEB 页面装修或复制第二套 B 商家线上商城。
- 使用 CRMEB `user_spread`、`user_brokerage`、`user_bill`、`now_money` 或分销体系承载本项目奖励。
- 使用 CRMEB 用户余额承载 B 或 C 收益。
- 把线下会员或套餐伪造成总部商城订单。
- 把 B 商家采购订单和 C 端商城订单混用。
- 自动执行生产结算或公户转私户。

### 15.1 现有代码复用边界

- 现有 `yfth_customer_relation` 不是永久 B 商家归属的最终权威模型。
- 现有 referral candidate 固定 90 天，不适合作为新 active 关系直接复用。
- 现有 `ReferralRewardServices` 一次归因可能生成多个 rule item ledger，不符合新方案“每事件只取一个循环档位”。
- 现有 referral code 明文保存，不符合动态会员码哈希要求。
- 后续必须新建或扩展独立业务模型，不能只改旧字段含义；可以复用审计、幂等、快照、冲正、随机码、哈希和行锁设计模式。
- 不得复用预约核销表承载会员码，不得把线上 5980 套餐改造成线下套餐成交。
- 不得删除现有稳定线上套餐、十个月权益、月度履约、产品额度和旧推荐台账代码；旧功能暂停新入口还是保留兼容，由后续开发阶段根据真实入口调查决定。
