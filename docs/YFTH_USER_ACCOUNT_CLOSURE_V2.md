# 御方通和用户销户 V2

## 目标

用户销户 V2 以“账号不可恢复、可重新注册、历史不继承”为核心语义，替换 V1 的全库 UID 列扫描方案。用户主动销户和总部代销户共用同一业务服务、同一门禁和同一事务，不建立第二套用户体系。

销户成功后：

- 原账号、手机号和微信身份从旧账号解绑，原账号立即不能登录；
- 所有已知 Token、会话和用户缓存立即撤销；
- 原手机号、账号、openid 和 unionid 可以注册或绑定到新的 UID；
- 新 UID 不继承旧订单、套餐会员、门店归属、推荐关系、奖励、角色或招商身份；
- 旧账号不可恢复；必要历史只通过随机、不可逆的销户主体编号关联。

## 用户流程

1. 用户阅读销户说明并执行预检。
2. 页面列出必须先处理的阻塞项，以及积分、优惠券等将被放弃的权益。
3. 用户通过密码或现有短信验证码完成账号安全验证。
4. 用户勾选账号注销协议，并输入精确确认短语 `确认注销`。
5. 服务在事务内锁定用户、重新预检、处理各业务域、写入可靠审计，再使旧账号失效。
6. 事务提交后立即撤销 Token 和缓存，客户端清除登录状态并返回未登录首页。

总部代销户使用同一核心服务；仅总部范围且具备独立权限的管理员可操作，必须填写不少于四个字的原因并完成二次确认。总部不能绕过资金、订单、合同或门店责任门禁。需要紧急停止访问时，应使用冻结或停用，不得伪装成销户。

## 数据处理矩阵

| 数据类别 | 处理方式 | 结果 |
| --- | --- | --- |
| 登录账号、密码、手机号、昵称、头像、openid、unionid | 删除或不可逆去标识 | 旧凭据不能登录，原标识可供新 UID 注册 |
| 地址、发票、收藏、购物车、足迹、搜索、优惠券、标签、通知、客服关系 | 删除 | 新账号不可见且不继承 |
| 当前会员、归属、推荐、邀请码、门店角色、门店客户关系 | 删除或关闭当前投影 | 旧账号不再拥有当前能力 |
| 店长、店员角色 | 自动撤销 | 不影响其他用户或门店 |
| 商城订单、支付、退款、售后、财务账单 | 匿名保留 | 保留交易事实，不再关联有效 UID 或直接身份信息 |
| 套餐购买、激活、权益履约、预约、核销、配送 | 匿名保留 | 保留履约事实，新账号不继承 |
| 奖励候选、线下结算 | 匿名保留 | 保留结算事实，不写入新账号 |
| 加盟申请、合同、付款凭证、筹备、验收 | 匿名保留 | 业务记录仅由随机销户主体编号关联 |
| 招商合伙人业绩、收益、等级事件 | 匿名保留 | 当前身份与邀请入口删除，历史事实保留 |
| 审计、幂等、权威事件 | 匿名保留 | 清除手机号、姓名、openid、unionid 和原 UID，只保留必要业务事实 |
| 只属于该账号的私人附件引用 | 解除引用 | 不再可由账号或普通业务页面访问；共享或合规归档文件不做不安全的物理误删 |

匿名历史写入 `yfth_account_closure_history_link`，只保存业务表、业务记录、随机销户主体编号和处理时间，不保存原 UID、手机号、账号或微信标识。`yfth_account_closure_subject` 保存销户编号、来源、操作人、原因、处理统计和审计状态，同样不保存可直接识别用户的信息。

## 业务门禁

以下任一情况存在时，用户和总部都不能直接完成销户：

- 待付款、待发货、待收货、售后中订单或退款处理中；
- 商城余额不为零；
- 奖励未结算、争议中或待处理；
- 预约、套餐权益或配送尚未履约；
- 正式门店负责人或招商合伙人责任尚未转移或终止；
- 加盟合同、付款、财务或开店事项仍待处理；
- 账号处于争议、司法、审计或风险冻结状态。

积分、优惠券等无现金价值权益会在预检中明确展示，由用户确认放弃后清除。

## 会员、推荐与门店

- 删除永久会员当前投影、可用推广码和邀请入口；
- 关闭涉及销户用户的 active 一级推荐关系，原因记录为 `account_closed`；
- C1 销户时，其下级用户保留原 B1 门店归属，但不再为已销户 C1 产生新收益；
- C2 销户时，关闭其当前推荐关系和当前归属投影；
- 店长、店员角色自动撤销；
- 正式门店负责人或招商合伙人必须先完成责任转移或终止。

## 事务、幂等与会话

服务使用明确的业务域处理清单，不再查询 `information_schema` 扫描所有 UID 形态列。执行流程为：事务内锁用户、复检门禁、创建随机销户主体、关闭当前关系、匿名化保留历史、删除账号态数据、写统一审计、失效旧用户行。任一步失败均整体回滚。

重复请求安全返回“账号已不存在或已注销”。旧 Token 在销户事务提交后立即被索引撤销，同时认证层还会校验数据库中的用户有效状态，因此即使 Redis 暂时不可用，旧 Token 也不能继续访问。

## 验证

隔离 MySQL Community 8.0.46 已验证：

- 用户主动销户与总部代销户；
- 密码、协议、确认短语和总部原因校验；
- 订单、退款、余额、奖励、履约、门店/合伙人责任、加盟事项和风险冻结门禁；
- 店长/店员撤销、C1/C2 推荐关系关闭和 C2 的 B1 归属保留；
- 商城、套餐、履约、奖励、加盟与招商历史匿名保留；
- Token 立即失效；
- 原手机号、账号、openid、unionid 绑定新 UID；
- 新账号不继承旧订单、会员、归属、推荐、奖励或角色；
- 重复销户幂等；
- 强制审计失败时事务整体回滚；
- migration run、定向 rollback、rerun。

PHP 7.4 语法、销户 V2 contract、相关身份 contract、Admin production build、H5 production build、mp-weixin production compile 和 `git diff --check` 均作为本轮合并门禁。生产部署不得用真实用户做破坏性验证，只允许专用 TEST 账号。
